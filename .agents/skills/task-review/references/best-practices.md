# task-review のベストプラクティス

## 目的

- 問題の有無を見極める
- 見つけた指摘事項を優先度順に整理する
- 次の修正判断につながる形で結果を残す

## 基本原則

- まず正本との整合を見る
- 指摘事項を要約より先に置く
- 重大度と根拠をセットで示す
- 問題なしでも残留リスクを明示する
- ユーザー判断が必要な確認は `request_user_input` ツールで残す

## 推奨フロー

1. `request_user_input` ツールで `goal`、`execution_mode`、対象範囲を定義する
2. 正本とレビュー対象を読み比べる
3. 指摘事項候補を集める
4. 根拠と重大度で絞る
5. 次アクションを提案する

## execution_mode の使い分け

### agent-led

- レビュー観点と停止条件を先に共有してから自律的に進める
- 高リスクな指摘事項や対象認識の差異だけ `request_user_input` ツールで確認する

### collab-led

- 観点の優先順位と finding の扱いを節目ごとに確認する
- 観点の優先順位と指摘事項の扱いを節目ごとに確認する
- 指摘採否に迷う点は `request_user_input` ツールで早めに合意を取る

## 良い成果物の条件

- 指摘事項が短く明確
- 根拠が参照可能
- 重大度の説明がある
- 次の task につながる

## 停止して確認すべきケース

- 想定外のレビュー対象が出てきた
- 仕様の解釈が割れる
- セキュリティや権限に関わる問題を見つけた
- 修正方針まで含めて合意が必要
