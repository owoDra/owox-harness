# 技術スタック更新のベストプラクティス

## 何を載せるか

- 継続利用する技術カテゴリ
- 現在採用している主要スタック
- 運用上意味のあるバージョン方針
- 採用判断を説明する ADR 参照

載せないほうがよい例:

- 一時的な検証用ライブラリ
- 個人のローカル環境依存ツール
- 採用判断が未整理で、ADR も存在しない項目

## テーブル設計

- 列は `技術名` / `採用スタック` / `バージョン` / `ADR参照` を固定する
- `技術名` は「UIフレームワーク」「バックエンド言語」のような役割名にする
- `採用スタック` は「React」「Go」「PostgreSQL」のような具体名にする
- `ADR参照` は現行判断へたどれる形で書く

例:

| 技術名 | 採用スタック | バージョン | ADR参照 |
| --- | --- | --- | --- |
| UIフレームワーク | React | 19.x.x | [ADR-003](../adr/ADR-003-react.md) |

## simple と monorepo の分け方

- `simple` は `# 技術スタック` の下に単一テーブルを置く
- `monorepo` は `## 共通` を先頭に置き、共通採用技術をまとめる
- `monorepo` のサブプロジェクト固有技術は `## <subproject>` ごとに分ける
- 共通技術とサブプロジェクト固有技術を同じ節に混在させない

## ADR 連携

- 技術スタック追加時は必ず `docs-update-adr` で ADR を作る
- 単なるバージョン更新でも、採用方針変更なら ADR 改訂を検討する
- ADR なしの行を残さない

## 更新時の作法

- lockfile や manifest とズレていないか確認する
- 同じ役割の重複行は統合を検討する
- バージョン表記は表内で粒度を揃える
- 採用中止した技術は、履歴を tech-stack に残し続けるより ADR 側で判断履歴を追えるようにする
