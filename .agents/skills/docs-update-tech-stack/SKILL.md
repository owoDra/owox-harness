## 目的

技術スタックの採用状況を、責務単位で見返しやすい表として保守し、各採用判断を ADR と結び付ける。

## 前提資料

- `docs/project/index.md` と関連資料を参照して、採用技術の文脈、利用言語、責務チーム、外部依存、構成を把握する
- `docs/project/glossary.md` を参照して技術名・採用スタック名・セクション名の命名を統一する
- `docs/project/tech-stack.md` を参照して既存の構成と表記を把握する
- `docs/project/adr/index.md` を参照して既存の ADR 参照形式を把握する
- `docs-update-adr` スキルを参照して技術スタック追加時に必要な ADR 作成手順を把握する
- `./references/best-practices.md` を参照して表の設計と更新ルールを把握する
- 関連する requirement / spec / ADR / task / lockfile / manifest を参照して実際の採用状態を確認する

## 前提知識

- 技術スタック資料は「何を採用しているか」を一覧化する正本であり、採用理由の正本は ADR に置く
- 技術スタックの追加・採用変更では、必ず `docs-update-adr` を使って対応する ADR を作成または更新し、その参照を記録する
- 1 行は 1 つの採用対象を表し、役割・採用品目・バージョン・判断根拠を分離する

## やること

1. `request_user_input` で、追加または更新したい技術名、採用スタック、バージョン、対象範囲、関連 ADR の有無を確認する
2. 現在の構成を確認し、資料構成を決める
   - 単一構成: `# 技術スタック` の下に単一の表を置く
   - 複数サブプロジェクト構成: `## 共通` セクションと、各サブプロジェクトごとの `## <subproject>` セクションを置く
3. manifest / lockfile / 設定ファイル / 既存プロジェクト資料を確認し、実際の採用状態と表記揺れを調べる
4. 追加または変更する採用判断について、対応する ADR が存在するか確認する
   - 存在しない場合は `docs-update-adr` を使って ADR を作成する
   - 既存 ADR で足りない場合は改訂し、現在の判断を参照できる状態にする
5. `docs/project/tech-stack.md` を更新する
   - 列順は `| 技術名 | 採用スタック | バージョン | ADR参照 |` で固定する
   - 単一構成の場合は単一テーブルを維持する
   - 複数サブプロジェクト構成の場合は `## 共通` と各 `## <subproject>` セクションごとに同じ列構成の表を置く
   - 行は役割単位で追加し、同じ役割に複数候補を詰め込まない
6. バージョンを記録する
   - 固定バージョンでなく方針だけ決まっている場合は `19.x.x` のように方針が読める形で書く
   - 未確定なら空欄にせず、`未確定` または合意済みの表記で状態を明示する
7. `ADR参照` 列に対応する ADR を記録する
   - 技術スタック追加時は必ず ADR を新規作成し、その参照を記録する
   - 既存技術の更新でも、現行判断の主参照がわかる形で書く
8. 既存行の統合・分割が必要なら、表全体の責務が崩れないように整理する

## ルール

- 技術スタックを追加するときは必ず ADR を作成し、`ADR参照` を埋める
- 単一構成と複数サブプロジェクト構成で見出し構造を混在させない
- 列順は `技術名` / `採用スタック` / `バージョン` / `ADR参照` で固定する
- 技術名は役割を書く。製品名だけを書かない
- 採用スタックは具体的な製品・サービス・ライブラリ名を書く
- バージョンは現行運用に必要な粒度で揃える

## 確認事項

- `request_user_input` または同等の確認で更新対象と対象範囲を確認した
- 技術スタック追加時に対応する ADR を作成または確認した
- 構成に応じた表構成になっている
- すべての行に `ADR参照` がある
- 技術名と採用スタックの役割分担が崩れていない
