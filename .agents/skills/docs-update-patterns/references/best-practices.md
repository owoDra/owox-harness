# pattern 更新のベストプラクティス

## 何を書くか

- 複数箇所で再利用したい共通パターン
- どこで適用するか
- 適用条件と例外条件
- 根拠となる正本

## 書かないほうがよいもの

- 単一機能だけのローカル事情
- 実装手順の全文
- 一時的な作業メモ

## 良い pattern の条件

- 複数の spec や実装で再利用できる
- 適用範囲と非適用範囲が明確である
- 例外条件が明示されている
- 関連 requirement、spec、ADR と接続している

## 更新時の作法

- 既存 pattern と重複しないか確認する
- 関連する requirement、spec、ADR、validation、実装との整合を確認する
- pattern 追加後の追従更新要否を確認する

## pattern で落としやすい観点

- 適用条件と非適用条件の切り分け
- 例外を許す条件
- ローカル都合と横断パターンの区別
- 実装例を置きすぎて手順書化しないこと
