# Pattern 更新のベストプラクティス

## 何を書くか

- 複数箇所で再利用したいコード実装、データ構造・設計、UI/UX 設計の共通パターン
- どこで適用し、どこでは適用しないか
- 適用条件、例外条件、逸脱が許される条件
- 根拠となる requirement、spec、ADR、運用上の理由
- representative な例

## 書かないほうがよいもの

- 単一機能だけで閉じるローカル事情
- 具体的な実装手順の全文
- 一時的な作業メモ

## 良い pattern の条件

- 複数の spec や実装で再利用できる
- 適用範囲と非適用範囲が明確である
- 例外や逸脱条件が明示されている
- 既存の requirement、spec、ADR と接続している
- レビューや実装時の判断基準として使える
- カテゴリとファイル名プレフィックスが一貫している

## 命名規則

- pattern のファイル名は `<prefix><slug>.md` 形式にする
- 実装系は `impl-`
- データ系は `data-`
- UI 系は `ui-`
- UX 系は `ux-`
- API 系は `api-`
- テスト系は `test-`
- 運用系は `ops-`
- `architecture` を使う場合も、実際に主対象となるプレフィックスを優先する

## 更新時の作法

- 既存 pattern を先に確認し、重複を避ける
- 関連する requirement、spec、ADR、validation、実装を見て矛盾がないか確認する
- pattern を追加したら、関連する spec、implementation、review の追従要否を確認する
- 再利用の見込みが低いものは無理に pattern 化しない
