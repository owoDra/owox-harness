# proposal 更新のベストプラクティス

## 何を書くか

- 背景
- 提案内容
- 代替案
- 利点
- リスク
- 未確定事項
- 正式化先候補

## 書かないほうがよいもの

- 採用済み仕様の確定記述
- 単なる検討メモの羅列
- 結論のない思いつきだけの案

## 良い proposal の条件

- 何を解決したいかが明確
- 代替案比較がある
- 採用した場合の昇格先が見える
- リスクと未確定事項が隠れていない

## 提案本文で分けるべきもの

- 背景と提案内容
- 提案内容と代替案
- 利点とリスク
- 現時点の結論と未確定事項

## 停止して確認すべきケース

- 提案対象の境界が requirement か architecture か曖昧
- 採用時に互換性を壊す可能性がある
- 代替案比較が不足している
- proposal ではなく ADR を先に切るべき判断が混ざっている

## 更新時の作法

- 既存 proposal と重複しないか確認する
- 採用後の昇格先を最初から意識する
- 採用済みで役目を終えたものは archive へ移す
- active と archive を混在させない
- 採用に必要な判断材料が不足している場合は research を先に補う
