# task-discovery のベストプラクティス

## 目的

- 現状理解をそろえる
- 次の意思決定に必要な不確実性を減らす
- 実装や資料更新の前に、影響範囲と論点を見える化する

## 基本原則

- まず正本を読む
- 調査対象と対象外を先に切る
- 事実と仮説を分ける
- 完了条件を先に決める
- ユーザー判断が必要な確認は `request_user_input` ツールで残す
- 次アクションに接続できる形で終える

## 推奨フロー

1. `request_user_input` ツールで調査の目的、`execution_mode`、完了条件を定義する
2. 対象の requirement / spec / ADR / code / test / docs を洗い出す
3. 正本から順に読む
4. 必要最小限の検証で現状を確かめる
5. 影響範囲と未解決事項を整理する
6. 次に進むべき task を提案する

## execution_mode の使い分け

### agent-led

- 調査計画と停止条件を最初に共有してから自律的に進める
- 人手確認は高リスク判断、前提変更、権限外変更時に限定する
- 確認が必要になったら `request_user_input` ツールで論点を絞って聞く
- 広く探索しすぎず、完了条件に達したら止める

### collab-led

- 論点の選定と優先順位づけを人間と一緒に行う
- 節目ごとに途中結果を見せて、調査の深さを調整する
- 節目の確認は `request_user_input` ツールで明示的に残す
- 不確実性が高い論点は早めに確認する

## 良い成果物の条件

- 何が分かったかが短く読める
- 何がまだ分からないかが明確
- どこに影響するかが列挙されている
- 次に何をすべきかが提案されている

## 停止して確認すべきケース

- 初期依頼の理解と異なる前提が見つかった
- 複数の解釈があり、後続作業に大きく影響する
- 変更禁止範囲に触れる可能性がある
- 想定以上に影響範囲が広い
- 機密、権限、セキュリティに関わる懸念がある
