# task-discovery のベストプラクティス

## 目的

- 現状理解をそろえる
- 次の意思決定に必要な不確実性を減らす
- 実装や更新前に影響範囲を見える化する

## 基本原則

- まず正本を読む
- 調査対象と対象外を先に切る
- 事実と仮説を分ける
- 完了条件を先に決める
- 広く探しすぎず、次の判断に必要な深さで止める
- 参照した資料と未参照の資料を分けて残す
- 次アクションに接続できる形で終える

## 推奨フロー

1. 調査対象、対象外、停止条件を定義する
2. requirement、spec、ADR、code、test、docs の順に現状を確認する
3. 必要最小限の再現や検証で事実を確かめる
4. 影響範囲と未解決事項を整理する
5. 次に進むべき task を提案する

## 停止して確認すべきケース

- 初期依頼の理解と異なる前提が見つかった
- 複数の解釈があり後続作業に大きく影響する
- 変更禁止範囲に触れる可能性がある
- 想定以上に影響範囲が広い
- 機密、権限、セキュリティに関わる懸念がある

## 良い成果物の条件

- 何が分かったかが短く読める
- 何がまだ分からないかが明確
- どこに影響するかが列挙されている
- 次に何をすべきかが示されている
