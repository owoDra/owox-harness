# Requirement 更新のベストプラクティス

## 何を書くか

- 何を実現するか
- なぜ必要か
- 何が成功か
- 何をやらないか
- どんな制約があるか

## 書かないほうがよいもの

- 実装手段の細部
- テスト手順の詳細
- 一回限りの作業メモ

## 良い requirement の条件

- **目標** が 1 文で説明できる
- **問題** が背景と必要性を示している
- **成功条件** が観測可能
- **やらないこと** が境界を明確にしている
- **制約** が後続の spec や実装に効く

## 更新時の作法

- requirement を最初の更新点にする
- 重複要求は統合を検討する
- 要求変更後に必要な spec / ADR / テスト追従を確認する
