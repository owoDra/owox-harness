# Requirement 更新のベストプラクティス

## 何を書くか

- 何を実現するか
- 誰のどんな課題を解くか
- なぜ今必要かを示す根拠
- 何が成功かを示す測定可能な指標
- どこまでを対象にするか
- 今回やらないこと
- どんな制約、品質特性、依存関係、リスクがあるか

## 書かないほうがよいもの

- 実装手段の細部
- テスト手順の詳細
- 一回限りの作業メモ

## 良い requirement の条件

- **目標** が 1 文で説明できる
- **ユーザー / 事業ニーズ** が要求の必要性と対象を示している
- **根拠** が要求の妥当性を追跡できる
- **成功指標** が観測可能で、必要なら基準値と目標値を持つ
- **対象範囲** と **対象外** が境界を明確にしている
- **制約 / 品質特性** が後続の spec や実装に効く
- **前提 / 依存関係 / リスク** が判断保留や他チーム依存を見落とさない

## 更新時の作法

- requirement を最初の更新点にする
- `project.kind` を確認し、`simple` は `docs/project/requirements/` 直下、`monorepo` は `docs/project/requirements/shared/` または `docs/project/requirements/<subproject>/` に配置する
- 重複要求は統合を検討する
- 既存 requirement、spec、ADR、validation と照合し、重複や矛盾がないか確認する
- 要求変更後に必要な spec / ADR / テスト追従を確認する
- requirement は成果と境界の正本に留め、詳細な振る舞いや手順は spec や validation に委譲する
