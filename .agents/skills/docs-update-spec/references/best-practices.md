# Spec 更新のベストプラクティス

## 何を書くか

- requirement を満たす具体的な振る舞い
- 入力と出力の契約
- 状態遷移や不変条件
- エラーや失敗時の扱い
- 認可、監査、性能、冪等性、ページングなどの横断ルール
- 互換性や非推奨化の扱い
- 関連 requirement と検証観点

## 書かないほうがよいもの

- 要求の背景説明の大半
- コードレベルの実装手順
- 一時的な開発メモ

## 良い spec の条件

- **概要** で対象範囲がわかる
- **入力** と **出力** が契約として読める
- **挙動** が観測可能な振る舞いとして書かれている
- **状態遷移 / 不変条件** が明確で矛盾しない
- **エラー / 例外** が実際の失敗条件と利用者側の期待動作に接続している
- **横断ルール** が認可、監査、性能、互換性などの共通条件を取りこぼさない
- **例** が代表的な通常系、異常系、境界条件を補えている
- **検証観点** が validation やテストへ接続できる

## 更新時の作法

- 関連 requirement を明示する
- `project.kind` を確認し、`simple` は `.agents/specs/` 直下、`monorepo` は `.agents/specs/shared/` または `.agents/specs/<subproject>/` に配置する
- 既存 pattern を確認し、局所 spec に閉じるより pattern 化した方がよい共通要素がないか確認する
- 既存 spec、requirement、ADR、validation、実装コード、テストコードと照合し、重複や矛盾がないか確認する
- 実装コードとテストコードの見直しをセットにする
- detail なテスト手順は validation に委譲し、spec には検証観点を残す
- architecture や validation と矛盾がないか確認する
