# Spec 更新のベストプラクティス

## 何を書くか

- requirement を満たす具体的な振る舞い
- 状態遷移や状態条件
- エラーや失敗時の扱い
- 関連 requirement

## 書かないほうがよいもの

- 要求の背景説明の大半
- コードレベルの実装手順
- 一時的な開発メモ

## 良い spec の条件

- **概要** で対象範囲がわかる
- **挙動** が観測可能な振る舞いとして書かれている
- **状態管理** が明確で矛盾しない
- **エラー** が実際の失敗条件に接続している
- **規約** が明確で矛盾しない
- **テストケース** で品質を担保できるだけの網羅度がある

## 更新時の作法

- 関連 requirement を明示する
- 実装コードとテストコードの見直しをセットにする
- architecture や validation と矛盾がないか確認する
