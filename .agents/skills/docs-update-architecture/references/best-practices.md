# architecture 更新のベストプラクティス

## 何を書くか

- 全体で守る不変条件
- 責務分離の原則
- 長期的に維持したい設計方針

## 書かないほうがよいもの

- チーム固有の細かな運用
- 一回限りの作業手順
- 個別機能だけの挙動仕様
- ハーネスによって決まるルール

## 更新時の必須追従

- spec を見直す
- code と test を見直す
- validation を見直す
- 必要なら ADR と tech-stack も更新する

## 良い architecture の条件

- 何を絶対に守るかが明確
- 責務境界が資料間で矛盾しない
- 実装と検証の期待に接続できる
- 変更時に追従範囲が判断できる

## architecture で落としやすい観点

- team guide に書くべき内容を持ち込んでいないか
- 個別機能仕様を architecture に書いていないか
- 原則変更後の validation 追従を忘れていないか
- directory 構造と責務分離が食い違っていないか
