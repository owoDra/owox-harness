# 用語集更新のベストプラクティス

## いつ追加するか

- プロジェクト内で繰り返し登場する用語
- 一般語だが、このプロジェクトでは独自の意味を持つ用語
- 略称・別名・表記揺れが発生しやすい用語
- requirement / spec / ADR の読み解きに必要な用語

追加しないほうがよい例:

- 一度しか出ない一時的な作業メモ
- コードのローカル変数名のような狭すぎる名前
- 一般辞書で十分で、このプロジェクト独自の意味を持たない語

## 良い定義の条件

- 先頭文だけで「それは何か」がわかる
- 読み手の語彙に依存しすぎない
- 循環定義になっていない
- 実装詳細ではなく意味と境界を述べる
- 必要なら近い概念との違いを短く添える

悪い例:

- 「ADR は ADR のこと」
- 「Glossary はいろいろ大事なことを書く場所」

良い例:

- 「ADR は、重要な設計・実装・運用判断とその理由を記録する正本。」

## 正規名称の決め方

- 既存ハーネス資料で最も一貫して使われている名称を優先する
- 略称より正式名称を優先する
- 複数候補がある場合は、もっとも誤解が少なく将来も耐久しやすい名称を採る
- 別名や略称が残るなら、独立項目を増やすより正規名称の定義に吸収する

## 更新時の作法

- 新規項目追加より、既存項目の改善で済むかを先に確認する
- ファイル全体の大規模整形より、意味の改善を優先する
- 用語集で決めた名称が他資料と衝突するなら、他資料側の表記統一も検討する
