# glossary 更新のベストプラクティス

## いつ追加するか

- 繰り返し登場する用語
- このプロジェクトで独自の意味を持つ用語
- 略称、別名、表記揺れが発生しやすい用語

## 良い定義の条件

- 先頭文だけでそれが何か分かる
- 循環定義になっていない
- 実装詳細ではなく意味と境界を述べる

## 正規名称の決め方

- 既存資料で最も一貫して使われている名称を優先する
- 略称より正式名称を優先する
- 別名や略称は独立項目より正規名称へ寄せる

## 更新時の作法

- 新規追加より既存項目の改善で済むかを先に確認する
- 用語集で決めた名称が他資料と衝突するなら他資料側も統一する

## glossary で落としやすい観点

- 一般語の一般的意味とプロジェクト固有意味の区別
- 略称を正規名称へ寄せる判断
- 用語の境界説明が足りず誤解を生むこと
- 実装詳細を定義文に混ぜること
