# task-validation のベストプラクティス

## 目的

- 完了条件を満たしているかを確認する
- 検証結果を証跡付きで残す
- 未達と未確認を明示する

## 基本原則

- まず validation と完了条件を確認する
- 合格条件を先に定義する
- 証跡を残せる検証を優先する
- 未確認を pass と見なさない
- ユーザー判断が必要な確認は `request_user_input` ツールで残す

## 推奨フロー

1. `request_user_input` ツールで `goal`、`execution_mode`、合格条件を定義する
2. 検証項目と証跡を決める
3. 実施する
4. 結果を項目ごとに整理する
5. 未達と次アクションを明示する

## execution_mode の使い分け

### agent-led

- 検証計画と停止条件を先に共有してから自律的に進める
- 重大な未達や合格条件の解釈差だけ `request_user_input` ツールで確認する

### collab-led

- 検証項目、証跡、未達時の扱いを節目ごとに確認する
- 判定に迷う点は `request_user_input` ツールで明示的に合意を取る

## 良い成果物の条件

- 項目ごとに結果が分かる
- 証跡が追える
- 未達と未確認が分かれている
- 次の修正や資料更新に接続できる

## 停止して確認すべきケース

- 合格条件の解釈が割れる
- 重大な未達がある
- 証跡が不足している
- validation 自体の不足が見つかった
