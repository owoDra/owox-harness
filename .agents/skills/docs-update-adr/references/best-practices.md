# ADR 更新のベストプラクティス

## ADR にするべき判断

- 設計や運用に継続的な影響がある
- 代替案があり、選択理由を残す価値がある
- 将来、なぜそうしたかを説明する必要がある

## 良い ADR の条件

- 1 文で判断内容を言える
- 背景に制約と問題設定がある
- 代替案と不採用理由がある
- 結果と追従作業が読める

## 書かないほうがよいもの

- その場限りの軽微な修正
- 既存方針に従っただけの自明な実装
- 単なる作業履歴や会議メモ

## 更新時の作法

- 置換関係がある場合は新しい ADR で追えるようにする
- active と archive の配置を使い分ける
- requirement や spec と接続できる関連資料を残す

## ADR で落としやすい観点

- 何を決めたかが 1 文で言えるか
- 何を捨てたかとその理由が残っているか
- 結果として何を更新すべきかが読めるか
- 背景が単なる時系列ログになっていないか
