# ADR 更新のベストプラクティス

## ADR にするべき判断

- アーキテクチャや運用に継続的な影響がある
- 代替案が複数あり、選択理由を残す価値がある
- 将来の変更時に背景確認が必要になる
- requirement や spec だけでは「なぜ」が残りきらない

ADR にしないほうがよい例:

- その場限りの軽微な修正
- 既存方針に従っただけの自明な実装
- 単なる作業履歴や会議メモ

## 良い ADR の条件

- 1 文で判断内容を言える
- Context に制約と問題設定がある
- Decision に採用内容と理由がある
- Alternatives に比較対象がある
- Consequences に副作用と追従作業がある

## 採番と改訂

- ID は連番で採番する
- ファイル名は `ADR-<NNN>-<short-title>.md` のように検索しやすくする
- `simple` では `.agents/adr/` 直下に置く
- `monorepo` では `.agents/adr/<subproject>/` を作り、その中に置く
- サブプロジェクト共通の判断だけを `.agents/adr/` 直下に置く
- 過去判断をひっくり返す場合は、旧 ADR を黙って書き換えるより新 ADR で supersede 関係を示す
- index は ADR 作成と同時に更新する

## 書き方

- タイトルは判断内容を表す短い文にする
- Context には症状ではなく制約と背景を書く
- Alternatives には少なくとも主要な不採用案を含める
- Consequences では利点だけでなく負債や移行コストも書く
- 断定できない点は曖昧にぼかさず、前提や未解決事項として切り分ける
