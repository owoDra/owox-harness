# 実装タスクにおけるコメントのベストプラクティス

## 目的

- コメントを保守のための資産にする
- 正本、設計判断、テストとの対応関係をコード上から追えるようにする
- 言語やフレームワークが違っても一貫した判断基準を持つ

## 基本原則

- その言語で標準的なドキュメンテーションコメントを使う
- コメントは「何をしているか」より「なぜそうしているか」を優先する
- コードから明らかな説明は書かない
- コメントは変更時に一緒に更新する
- 読み手が次に見るべき正本、関連コード、関連テストへ辿れるようにする

## 書くべき内容

- この実装が満たす requirement や spec
- この実装が従う ADR や設計判断
- 守るべき不変条件
- 境界条件、例外条件、互換条件
- 重要な依存先や連携先
- 関連テストや検証経路

## 書かない方がよい内容

- 1 行ずつ処理をなぞる説明
- 変数代入や単純分岐の言い換え
- すでに名前や型で分かる説明
- 実装と同期されない TODO の羅列

## トレースタグの使い方

- `.agents/templates/trace-tags.md` のタグを使う
- ドキュメンテーションコメントのまとまりとして置く
- 推奨する最低限の組み合わせは以下
  - requirement に対応する実装: `@req`
  - 仕様や振る舞いに対応する実装: `@spec`
  - 重要な設計判断に依存する実装: `@adr`
  - 守るべき条件がある実装: `@invariant`
  - 実装理由を残す価値がある箇所: `@reason`
  - 関連テストや確認経路: `@test`

## 記載位置の指針

- モジュール全体の責務に関わるならモジュール先頭
- 型、クラス、公開関数に関わるなら定義直前
- 特定の分岐や設定値に関わるなら、その単位の直前
- 多用しすぎず、後から保守者が判断理由を追うために必要な位置へ置く

## 言語やフレームワークをまたぐ共通判断

- 公開 API や拡張ポイントは優先して記録する
- 外部システムとの境界は優先して記録する
- 暗黙制約やデータ整合性ルールは優先して記録する
- 性能最適化や回避策は理由を残す
- セキュリティ、権限、機密データの扱いは理由と不変条件を残す

## 良いコメントの条件

- 単体で読んでも意図が分かる
- 正本やテストへの入口になる
- 実装変更時に更新しやすい長さである
- 特定言語の記法に寄りすぎず意味が保たれている

## 避けるべき状態

- タグだけあって意味説明がない
- 正本参照が古い
- 実装位置と離れすぎている
- コメントが多すぎて本体の可読性を下げている
