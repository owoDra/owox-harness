# task-implementation のベストプラクティス

## 目的

- 正本に沿って実装する
- 変更を小さく保つ
- 検証と資料更新まで一貫して終える

## 基本原則

- まず正本を確認する
- 変更範囲と対象外を先に切る
- 実装前に検証方法を決める
- 実装と検証を分離しすぎない
- ユーザー判断が必要な確認は `request_user_input` ツールで残す

## 推奨フロー

1. `request_user_input` ツールで `goal`、`execution_mode`、完了条件を定義する
2. 正本と既存実装を確認する
3. 実装方針と検証方針を定める
4. 小さい単位で実装する
5. テストと整合確認を実行する
6. 必要な正本更新を行う
7. 完了条件に照らして閉じる

## execution_mode の使い分け

### agent-led

- 実装計画と停止条件を最初に共有してから自律的に進める
- 重要な方針変更、仕様変更、高リスク変更だけ `request_user_input` ツールで確認する
- 完了条件を満たしたら広げすぎずに止める

### collab-led

- 実装方針、主要変更点、検証方針を節目ごとに確認する
- 大きい変更に入る前に `request_user_input` ツールで合意を取る
- 実装の深さや対象範囲を都度調整する

## 良い成果物の条件

- 変更意図が正本に接続している
- 検証結果がある
- 必要な正本更新が終わっている
- 変更範囲が説明可能である

## 停止して確認すべきケース

- 正本に不足や矛盾がある
- 互換性を壊す可能性がある
- 想定より影響範囲が広い
- セキュリティ、権限、機密に関わる
- テスト方針の変更が必要である
