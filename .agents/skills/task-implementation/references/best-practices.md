# task-implementation のベストプラクティス

## 目的

- 正本に沿って実装する
- 変更を小さく保つ
- 検証と資料更新まで一貫して終える
- 互換性、移行、ロールアウト、観測可能性を実装の一部として扱う
- コメントとトレースタグを保守のための資産にする

## 基本原則

- まず正本を確認する
- 変更範囲と対象外を先に切る
- 現状挙動と変えたい挙動を分けて把握する
- 実装前に検証方法を決める
- 実装と検証を分離しすぎない
- 変更は小さい一貫した単位で行う
- 変更理由と正本との対応関係をコード上でも追跡できるようにする
- 外部公開契約、データ整合性、権限、性能への影響を先に確認する
- 既存 pattern を優先し、新しいローカル流儀を増やす前に pattern 化の要否を考える
- ユーザー判断が必要な確認は `request_user_input` ツールで残す

## 推奨フロー

1. `request_user_input` ツールで `goal`、`execution_mode`、完了条件を定義する
2. 正本、既存実装、既存テスト、既存の観測手段を確認し、現状挙動を把握する
3. 実装方針、互換性方針、移行有無、ロールアウト方針、検証方針、pattern 適用方針を定める
4. 小さい単位で実装し、必要箇所にトレースタグ付きコメントを残す
5. 変更に応じて単体、結合、E2E、手動確認、lint、型チェック、静的解析を実行する
6. 必要な正本更新を行う
7. 未実施検証と残リスクを整理する
8. 完了条件に照らして閉じる

## トレースタグ

- `./trace-tags.md` の形式に従う
- トレースタグはその言語で標準的に使われるドキュメンテーションコメントに記載する
- 少なくとも `@req`, `@spec`, `@test` は追えるようにし、重要判断がある場合は `@adr`, `@reason`, `@invariant` も付ける
- 変更理由を説明する価値がある単位に付ける。処理行ごとに乱用しない
- 関連コード参照が必要な場合は、コメント本文にモジュール名、型名、関数名、設定名などを明記して関係を追えるようにする

## コメントとドキュメンテーションコメント

- その言語で標準的なドキュメンテーションコメントを使う
- コメントはコードを読むだけでは分からない意図を書く
- 境界条件、失敗時の扱い、不変条件、互換性理由を優先して書く
- この実装が満たす requirement / spec / ADR を追えるようにする
- 重要な依存先、連携先、関連テストや検証経路への入口を残す
- 名前や型だけで読める処理説明はコメントにしない
- 1 行ずつ処理をなぞる説明は書かない
- タグだけ置いて意味説明がない状態を避ける
- 実装とズレた古いコメントを残さない
- コメントが本体の可読性を下げるほど増えないようにする

## execution_mode の使い分け

### agent-led

- 実装計画と停止条件を最初に共有してから自律的に進める
- 重要な方針変更、仕様変更、高リスク変更だけ `request_user_input` ツールで確認する
- 完了条件を満たしたら広げすぎずに止める

### collab-led

- 実装方針、主要変更点、検証方針を節目ごとに確認する
- 大きい変更に入る前に `request_user_input` ツールで合意を取る
- 実装の深さや対象範囲を都度調整する

## 良い成果物の条件

- 変更意図が正本に接続している
- 検証結果がある
- 未実施の検証と残リスクが把握できる
- 必要な正本更新が終わっている
- 変更範囲が説明可能である
- 重要な変更箇所から正本と関連テストを追える
- 互換性、移行、運用影響が説明可能である
- pattern 適用理由または pattern 不適用理由が説明可能である

## 停止して確認すべきケース

- 正本に不足や矛盾がある
- 互換性を壊す可能性がある
- 想定より影響範囲が広い
- データ移行やロールアウト手順が必要である
- セキュリティ、権限、機密に関わる
- 入力検証や外部境界の扱いが変わる
- 共通化すべきなのにローカルな流儀が増えている
- テスト方針の変更が必要である
