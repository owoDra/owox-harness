# task-implementation のベストプラクティス

## 基本原則

- まず正本を確認する
- 変更範囲と対象外を先に切る
- 現状挙動と変えたい挙動を分けて把握する
- 実装前に検証方法を決める
- 変更は小さい一貫した単位で行う
- 共通化すべき流儀は pattern 化を検討する
- 互換性、移行、ロールアウト、可観測性も実装の一部として扱う
- 正本不足があるなら決め打ち実装へ進まない

## 推奨フロー

1. 正本、既存実装、既存テストを確認する
2. 実装方針と検証方針を定める
3. 小さな単位で実装する
4. 変更に応じたテスト、lint、型チェック、手動確認を行う
5. 正本更新要否と残リスクを整理する

## コメントと trace tag

- コメントは処理説明ではなく、意図、不変条件、境界条件、根拠を書く
- trace tag は重要な関数、クラス、モジュール、設定に付ける
- trace tag には関連する requirement、spec、ADR、pattern、validation、test の参照を含める
- 変更理由を説明するうえで必要なら関連する docs/project 配下の正本へたどれるようにする
- 行ごとの乱用は避ける
- 実装とズレた古いタグやコメントを残さない

## 良い trace tag の条件

- 変更理由と正本の関係が追える
- 関連テストや検証経路にたどれる
- 意図や不変条件がタグの周辺説明だけで理解できる

## 停止して確認すべきケース

- 正本に不足や矛盾がある
- 互換性を壊す可能性がある
- 想定より影響範囲が広い
- データ移行やロールアウト手順が必要
- セキュリティ、権限、機密に関わる

## 良い成果物の条件

- 変更意図が正本に接続している
- 検証結果がある
- 必要な正本更新が終わっている
- 未実施の検証と残リスクが把握できる
