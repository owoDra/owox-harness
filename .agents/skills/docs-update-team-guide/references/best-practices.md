# Team Guide 更新のベストプラクティス

## 何を書くか

- そのチームが最終責任を持つ役割
- 他チームとの境界
- 継続的に守る作法とレビュー観点
- チーム固有の業務知識や運用知識

## 書かないほうがよいもの

- 全チーム共通の不変条件
- 一回限りのタスクメモ
- 特定実装の細部だけに閉じた説明

## 良い guide の条件

- チームの責務が 1 文で説明できる
- 境界が明確で、他チームとの受け渡しが読める
- ルールが行動に落ちる形で書かれている
- 固有知識が必要な人にだけ効く内容になっている

## 更新時の作法

- `project.teams` を基準に名称を合わせる
- 他資料と同じ用語を使う
- 境界が変わる場合は architecture や spec 側への影響も確認する
