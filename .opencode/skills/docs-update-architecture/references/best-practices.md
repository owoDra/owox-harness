# Architecture 更新のベストプラクティス

## 何を書くか

- 全体で守る不変条件
- 役割や責務の分離原則
- 長期的に維持したい設計理念

## 書かないほうがよいもの

- チーム固有の細かな運用
- 一回限りの作業手順
- 個別機能だけの挙動仕様
- ハーネスによって決まっているルールやふるまい

## 変更時の必須追従

- specs をすべて見直す
- 実装コードをすべて見直す
- テスト用コードをすべて見直す
- validation を見直す
- 必要なら ADR と技術スタックも更新する

## 良い architecture の条件

- 何を絶対に守るかが明確
- 責務境界が資料間で矛盾しない
- 実装と検証の期待に接続できる
- 変更時に追従範囲が判断できる
