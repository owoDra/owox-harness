# task-review のベストプラクティス

## 基本原則

- まず正本との整合を見る
- 既存 pattern への準拠と pattern 化不足を見る
- 指摘事項を要約より先に置く
- 重大度と根拠をセットで示す
- 問題なしの場合でも残留リスクや検証不足を明示する

## 推奨フロー

1. 対象、対象外、優先観点を定義する
2. 正本とレビュー対象を読み比べる
3. 指摘候補を集め、根拠と重大度で絞る
4. 次アクションを提案する

## 停止して確認すべきケース

- 想定外のレビュー対象が出てきた
- 仕様の解釈が割れる
- セキュリティや権限に関わる問題を見つけた
- 修正方針まで含めて合意が必要

## 良い成果物の条件

- 指摘事項が短く明確
- 根拠が参照できる
- 重大度の説明がある
- 次の task につながる
