# task-fix のベストプラクティス

## 目的

- 症状を正しく捉える
- 原因を特定する
- 修正後の回帰を防ぐ

## 基本原則

- まず症状を確認する
- 症状と原因を混同しない
- 修正前に回帰確認方法を決める
- 仕様問題は仕様として扱う
- ユーザー判断が必要な確認は `question` ツールで残す

## 推奨フロー

1. `question` ツールで `goal`、`execution_mode`、完了条件を定義する
2. 症状を再現または確認する
3. 原因候補を絞る
4. 最小修正で直す
5. 回帰確認を実行する
6. 必要な正本更新を行う

## execution_mode の使い分け

### agent-led

- 再現方針と停止条件を先に共有してから自律的に進める
- 原因の想定変更や高リスク修正だけ `question` ツールで確認する

### collab-led

- 再現結果、原因仮説、修正方針を節目ごとに確認する
- 互換性や優先順位に関わる判断は `question` ツールで合意を取る

## 良い成果物の条件

- 症状、原因、修正、回帰確認が分かれている
- 修正が最小限である
- 再発防止観点がある
- 必要なら正本が更新されている

## 停止して確認すべきケース

- 再現できない
- 根本原因が複数あり得る
- 修正に互換性リスクがある
- 仕様変更が必要
