# Integration 資料更新のベストプラクティス

## 何を書くか

- その外部 API・サービスの役割
- どこで使うか、どこまで依存するか
- 認証・権限の考え方
- 制約、レート制限、運用注意点
- 障害時の扱いと関連資料

## 書かないほうがよいもの

- 実際の秘密情報
- 一回限りの手動操作メモ
- 実装コード断片だけの説明

## 良い integration 資料の条件

- その依存が何のためにあるかすぐわかる
- 誰が責任を持つか読める
- 認証や制約が運用に使える粒度で書かれている
- 障害時の扱いと関連資料にたどれる

## 更新時の作法

- `project.integrations` の名称とそろえる
- 用語集の命名を優先する
- 制約変更があれば validation や ADR の更新要否も確認する
- 採用技術としても重要なら tech-stack にも反映する
